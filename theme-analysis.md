# 规则即牢笼 — Game Jam 创意分析

> 主题：规则即牢笼。谁制定规则？谁被规则困住？在遵守、利用或打破规则之间，探索自由的可能。
> 工程：Godot 4.6+ Mono (C#)；greenfield（目录初始为空）。
> **当前采用方向：v2 爽游「牢笼崩坏」（竞技场动作）。** v1 解谜方向见附录（已弃用，保留参考）。

## 1. 主题解构

游戏本身就是一套规则系统，玩家天然处在「被规则困住」的位置——这是它作为元游戏主题的天然优势：

- **谁制定规则** → 系统的 authorship（关卡 / 叙事 / 引擎本身，或关内的「狱卒」）。
- **谁被规则困住** → 玩家与 NPC 的双重困境，暗含「共谋」：遵守规则的人也在维持牢笼。
- **遵守 / 利用 / 打破** → 玩家的 agency（能动性）。设计目标不是逼玩家选边，而是**让玩家先意识到规则的存在，再获得对规则的操作权**。

## 2. 三立场设计脊柱（动作化重映射）

同一套规则系统，三种互动方式，是整局游戏的脊柱；在动作游戏里从「解谜选项」变成「实时风险/收益」：

| 立场 | 动作里的映射 | 风险 | 奖励 |
|------|------|------|------|
| 遵守 Obey | 保守打法，不触发任何规则事件 | 低 | 安全推进 · 顺从结局 |
| 利用 Exploit | 不破规但借规反制（诱敌入禁区、推箱挡刀） | 中 | 战术优势 · 隐藏路径 |
| 打破 Break | 砸碎规则 totem，解锁能力 + 全场崩坏 | 高（受守护规则刷执法者） | 新能力 · 解放结局 |

**关键设计点**：记录玩家每关的立场选择，累计出主导立场 → 决定三种结局之一。
好处：低成本重玩性（同一关卡三种玩法），适合 48h Game Jam。

## 3. v2 方向：爽游「牢笼崩坏」（竞技场动作）★ 当前采用

**一句话**：竞技场动作。规则=限制器（牢笼），砸碎规则 totem → 立刻解锁能力 + 全场崩坏演出（震屏/慢镜/飘字）→ 连段清场，再推进到下一间更狠的牢笼。

**核心循环（5 拍，见文档外的循环图）**：受限开局 → 发现规则 → 砸碎规则 → 解锁·崩坏 → 连段清场 →（越狱推进，难度递增）回到受限开局。

**规则即限制器示例**（每条都能做成 `Rule` 资源 + 对应 `RuleTotem`）：
- `移动速度上限` → 打破后相位冲刺。
- `禁止近战` → 打破后获得巨剑。
- `禁止造成伤害` → 打破后全场 AOE。
- `重力锁定` → 打破后浮空连击。

**多巴胺设计**：每次打破触发 震屏 + 慢镜（hit-stop）+ 飘字 + 音乐强度升档——全部由 `RuleEngine` 信号驱动，UI/音频直接订阅，零轮询。这是「爽」的来源，也是动作游戏不能卡顿的硬指标。

## 4. Godot C# 架构（复用 v1 数据层 + 动作化）

- **Rule** : `Resource`（`[GlobalClass]`）— `StringName Id`、`string Description`、`bool Evaluate(RuleContext ctx)`。不变。
- **RuleContext** — 持有 actor 状态 / 位置 / 标志位，供谓词求值。不变。
- **RuleEngine**（`Autoload` 单例）— 持有活动规则，暴露 `IsViolated(...)`；发出 `RuleBroken` / `RuleExploited` / `RuleObeyed` 信号。**信号驱动，零轮询**。不变。
- **RuleTotem**（Node2D/3D）— 取代 `RuleSign`：可破坏、有 HP、带打破 VFX；被砸碎时向 `RuleEngine` 发 `RuleBroken` 信号并解锁对应能力。
- **Enforcer**（Node）— 打破「受守护规则」时由 `RuleEngine` 生成；实时追击协程用 `ToSignal`（单源等待，同步延续）驱动。
- **ComboDirector / 强度** — `RuleEngine` 统计打破数 → 缩放敌人密度 / 音乐强度，信号驱动音频。
- **RunState**（`Autoload`）— 记录三种立场计数，结算时选结局。不变。
- **UI / HUD** — 订阅 `RuleEngine` 信号做飘字、血条、立场条反馈；绝不轮询。

### 选型理由（异步 / 引用）
- `RuleEngine` 信号用 `+=` 订阅并配对 `-=`，或源释放时 Godot 自动清连接；自定义信号接收方可能先释放时优先 `Connect(SignalName.X, Callable.From(...))`。
- `Enforcer` 追击：单源等待用 `await ToSignal(...)`（信号当帧同步执行）；需取消 / 多源竞速才上 `TaskCompletionSource`。
- 节点引用：设计期节点用 `[Export]` 注入；仅运行时动态节点用 `GetNode<RuleTotem>("%Name")`（`unique_name_in_owner=true`）。硬编码路径一律禁止。
- **性能要点（爽游刚需）**：敌人 / 子弹走对象池；闲置节点 `ProcessMode = Disabled`；信号流零分配，避免 GC 卡顿破坏手感。

## 5. 推荐下一步（落地顺序，当前）

1. 锁定 **牢笼崩坏（竞技场动作）** 核心循环。
2. 先落地数据层：`Rule`（`[GlobalClass]` Resource）+ `RuleEngine`（Autoload）+ `RuleContext`（1 天内可跑）。
3. 做 `RuleTotem`（可破坏 + 打破 VFX）+ 玩家基础移动/攻击 + 最小可玩单间牢笼。
4. 加 `Enforcer` 实时追击 + `ComboDirector` 强度缩放（打破越多越炸）。
5. `RunState` 结局分支 + UI 信号反馈（飘字/震屏/音乐档位）。
